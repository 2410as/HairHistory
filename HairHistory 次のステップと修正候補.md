## 2. 要件・機能一覧とのギャップ（要判断）

- **パーマ**  
  - [要件定義](./HairHistory%20要件定義.md) の MVP には「パーマ」が含まれる。  
  - バックエンドの `ServiceType` には **パーマ専用コードがない**（`color` / `bleach` / `straight_perm` / `treatment` / `other` のみ）。  
  - **対応案**: `other`＋メモで運用／`perm` をドメイン・API・DB・フロントに追加。

- **施術の複数選択**  
  - 要件では「複数選択 or メモ」のニュアンス。  
  - 現状 API・UI は **1カテゴリ（1コード）** 前提。複数にするなら API の `services` 配列と UI（チップ複数選択）の両方を設計。

- **「施術記録」と「アーカイブ」**  
  - ナビで両方あるが、現状どちらも同一ページ内アンカー（`#archive-add`）または同一画面。  
  - 将来ルートを分けるなら、`/h/[id]` と別 URL の整理方針を決める。

- **匿名ユーザーの「初回」**  
  - 機能一覧は「初回アクセス時に ID 発行」寄りの表現。  
  - 実装はトップの「はじめる」で POST。仕様として揃えるならドキュメント側を更新。

## 4. バックエンド・インフラ

- **本番 URL**: `NEXT_PUBLIC_API_URL` とフロントのデプロイ先を一致させ、**HTTPS** を前提に CORS を設定。
- **DB バックアップ・復元**（要件の非機能）: 運用方針のドキュメント化。
- **レート制限・不正削除**: URL を知れば履歴が編集可能なモデルのままなら、最低限の注意書き（ポリシー）と、将来的なトークン保護の検討。

---

## 5. 品質・保守

- **E2E / 結合テスト**: 履歴の追加・編集・削除・404 リダイレクトの自動テスト。
- **アクセシビリティ**: フォーカスリング、コントラスト、`aria` の見直し（特にチップボタン・モーダル化した場合）。
- **デザイン統一**: ランディング（`landing-*`）とアーカイブ（`archive-*`）のタイポ・余白トークンを共通化するか方針決め。
