# HairHistory 技術選定

## 前提（要件から）

- **Webアプリ**（PC / スマホブラウザ）
- **ログインなし**（ゲスト利用）
- **ユーザーURL**で「自分の履歴」を一意に識別し、美容師と共有
- 履歴データ：日付・施術内容・サロン・スタイリスト・メモ（写真は扱わない）
- サロン・スタイリストはフリーテキスト
- --

## 1. フロントエンド

- **役割**: 画面表示、履歴の登録・編集・一覧、ユーザーURL用ページ
- **候補例**:

- React / Next.js

- Vue / Nuxt

- その他（Svelte, Astro など）

| 候補 | メモ |
|------|------|

| Next.js / React | 採用 |

- *方針**: **Next.js + React** で進める。
- --

## 2. バックエンド・データ保存

- **役割**: ユーザー（URL）の生成、履歴のCRUD、ユーザーと履歴の紐づけ
- **ログインなしで「ユーザー」を識別する方法**:

- 初回アクセス時にランダムID（ユーザーURL用）を発行し、localStorage などに保存

- そのIDがURLに含まれる（例: `https://example.com/h/abc123`）→ 同じIDなら同じ「自分の履歴」として扱う

- *API**: **Go** で自作（採用済み）

### 2.1 データベース（履歴・ユーザーID など）

| 候補 | メモ |
|------|------|

| **PostgreSQL** | Railway でプラグインとして追加可能。永続化が確実で、再デプロイでもデータが残る。Go からは pgx や GORM で接続。 |

| SQLite | 軽量だが、Railway はコンテナ再デプロイでファイルが消える可能性がある。永続化には Volume が必要。 |

- *方針**: **PostgreSQL** を Railway で追加して使う（シンプルかつ確実）。
- --

## 3. ホスティング・デプロイ

- **役割**: Next.js（フロント）と Go（API）の両方を公開、HTTPS、ユーザーURL が動作する環境
- **構成**: フロントとAPIを分離してデプロイする想定（Next.js 用 + Go 用の2つ）

| 候補 | メモ |
|------|------|

| **Vercel**（Next.js） + **Cloud Run**（Go） | Next.js は Vercel が最適。Go API は Cloud Run でコンテナ実行、スケールゼロでコスト抑えられる。 |

| **Vercel**（Next.js） + **Railway**（Go） | Railway は Go のデプロイが簡単。無料枠あり。 |

| **Railway**（Next.js も Go も） | 1つのプラットフォームで両方デプロイ。ダッシュボードが分かりやすく、MVP では運用が楽。 |

| **Vercel**（Next.js） + **Fly.io**（Go） | Fly.io は Go に強い。グローバル分散。 |

- *おすすめ**:
- **MVP・まず動かすなら**: **Railway** で Next.js と Go の両方をデプロイ。設定が少なく、1画面で両方見られる。
- **フロントとAPIを分けてスケールさせたいなら**: **Vercel（Next.js）+ Cloud Run（Go）**。Next.js は Vercel の無料枠で、Go は Cloud Run の従量課金で使える。
- *方針**: **Railway** で Next.js と Go の両方をデプロイする（シンプルに進める）。
